# Stage 1 Summary

In Stage 1, I built the first working version of my exoplanet transit detection project. My goal at this stage was to create a minimum viable research prototype that was realistic, understandable, and beginner-friendly. I organized the project into a clear structure with separate files for configuration, data loading, preprocessing, labeling, modeling, evaluation, visualization, and the main training pipeline. I wanted the code to be modular and easy to explain, so each part of the pipeline had its own responsibility.

For the data workflow, I designed the project to work with Kepler light-curve data, which records how a star's brightness changes over time. Since building a real labeled astronomy dataset is complicated, I made a simplifying assumption for the first version. I used real Kepler-style light curves as the background data, but I created labels by injecting simple synthetic transit-like dips into some windows of the data. That allowed me to create a supervised learning problem where some examples represented non-transit signals and others represented transit-like signals. This gave me a clean and understandable way to train the first model without depending on a difficult catalog-matching step right away.

For the model itself, I started with a logistic regression baseline because it is simple, fast, and easy to explain. I also extracted a small set of hand-crafted features from each light-curve window instead of using the raw signal directly. These features included values such as the minimum flux, the average and spread of the flux values, the depth of the dip, and how long unusually low values continued in a row. I then trained the model and evaluated it using accuracy, precision, recall, F1 score, and a confusion matrix. I also saved plots showing example light curves, labeled windows, and model results so the system was easier to inspect visually. By the end of Stage 1, I had a full end-to-end prototype that could load data, preprocess it, generate labels, train a baseline model, evaluate performance, and produce outputs I could analyze.

Later in Stage 1, I also began designing a simple local app for the project using Streamlit. This app allowed a user to upload a CSV file containing light-curve data and run the model as a transit-like signal screener. At first, the model was only learning during runtime, which meant it disappeared when the program ended. To fix that, I added model saving so the trained model could be stored on disk and reused later. That was an important improvement because it changed the project from a temporary in-memory prototype into something that could keep its learned state between runs.
